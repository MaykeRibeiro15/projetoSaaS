    public.rls_auto_enable() com SECURITY DEFINER exposta
    para anon e authenticated. ESSA FUNÇÃO NÃO FOI CRIADA
    NESTA SESSÃO — já existia previamente no projeto
    (provavelmente um helper antigo). NÃO foi mexida para
    não impactar nada existente. Avaliar manualmente:
        - REVOKE EXECUTE ON FUNCTION public.rls_auto_enable()
          FROM anon, authenticated;

      remediação:
        https://supabase.com/docs/guides/database/database-linter?lint=0028_anon_security_definer_function_executable

4. O QUE FALTOU / PRÓXIMOS PASSOS
============================================================

Pontos pendentes em ordem de prioridade:

  (1) Sincronizar com Prisma — IMPORTANTE
      O backend NestJS usa Prisma como source-of-truth do
      schema, mas as migrations criadas vivem APENAS no
      banco Supabase (via MCP). A pasta prisma/migrations/
      no repositório continua VAZIA.

      Sugestões (escolher uma):
        (a) Rodar localmente:
              cd backend
              npx prisma db pull
              npx prisma migrate diff \
                --from-empty \
                --to-schema-datamodel prisma/schema.prisma \
                --script > prisma/migrations/0_init/migration.sql
              npx prisma migrate resolve --applied "0_init"
            Isso versiona a baseline para evitar drift.
        (b) OU dropar tudo e refazer via "prisma migrate dev
            --name init" usando a DATABASE_URL do Supabase
            (mais limpo, mas apaga as 15 tabelas vazias e
            recria).

  (2) Configurar app.current_tenant_id na conexão Prisma
      Para a RLS funcionar quando NÃO usar service_role:
      criar Prisma middleware/extension que execute
        SET app.current_tenant_id = '<tenant_id_do_JWT>'
      em cada request autenticado. Hoje as policies negam
      tudo a quem não setar essa variável.

  (3) Service Role vs Anon
      O .env do backend precisa de SUPABASE_SERVICE_ROLE_KEY
      configurada se quiser bypass de RLS no NestJS. Não
      foi configurado nesta sessão.

  (4) Função pré-existente rls_auto_enable()
      Decidir REVOKE / DROP / converter para SECURITY
      INVOKER conforme o uso original.

  (5) Seed
      Continua faltando prisma/seed.ts (1 tenant + 1 doctor
      + 2 pacientes + 3 appointments) para QA local. Não
      foi criado nesta sessão.

  (6) Connection strings
      Não populei backend/.env com DATABASE_URL/DIRECT_URL
      do Supabase porque nenhuma credencial foi
      compartilhada. Quando for ligar o backend ao banco:
        - DATABASE_URL = pooler 6543 (pgbouncer transaction)
        - DIRECT_URL   = porta 5432 (migrações)
        - SUPABASE_URL = https://qakfcyurfroqjvykbdwe.supabase.co
        - SUPABASE_ANON_KEY = (pegar do painel)
        - SUPABASE_SERVICE_ROLE_KEY = (pegar do painel)

  (7) Storage / Edge Functions
      Não foi configurado bucket de Storage (necessário
